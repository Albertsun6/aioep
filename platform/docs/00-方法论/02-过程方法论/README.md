# 02 - 过程方法论

> 状态: **已确认** | 版本: v0.2.0 | 创建: 2026-02-15 | 更新: 2026-02-15

## 定位

定义项目"怎么推进"。整合 6 个过程方法论，明确每个方法论在哪个阶段主导、辅助或不参与。不追求完整覆盖每个方法论标准，只取与 AI 驱动企业应用开发相关的核心实践。

---

## 阶段定义（4 阶段模型）

> 原规划为 Phase 0~5（6 阶段），实际合并为 Phase 0~3（4 阶段），更适合 1 人 + AI 的项目规模。详见 [sdlc.md](./sdlc.md)。

| 阶段 | 名称 | 周期 | 核心活动 |
|------|------|------|---------|
| Phase 0 | 规划与准备 | — | 0A 方法论制定+项目定义+开发规范 / 0B Docker 环境+CI/CD |
| Phase 1 | 需求分析与设计 | 2~3 周 | 纯分析：BRD、Gap-Fit、BPMN、UML、RBAC、BDD（不写代码） |
| Phase 2 | 全部开发 | 10 周 | Sprint 0 erp_base → Sprint 1~3 核心模块+报表 |
| Phase 3 | 集成 + 上线 | 4 周 | 外部集成、UAT、数据迁移、部署、培训 |

## 方法论 x 阶段映射

| 方法论 | Phase 0 规划与准备 | Phase 1 需求分析 | Phase 2 全部开发 | Phase 3 集成+上线 |
|-------|:---:|:---:|:---:|:---:|
| TOGAF/ArchiMate | - | **主导** | 参考 | - |
| SDLC | 辅助 | **主导** | 框架 | **主导** |
| Agile/Scrum | - | - | **主导** | 辅助 |
| DevOps | **主导** | 辅助 | 辅助 | **主导** |
| ITIL | - | - | - | **主导** |
| Lean/Kanban | 辅助 | 辅助 | 辅助 | 辅助 |

**怎么读这个矩阵**：
- **主导**：该阶段的核心执行方式由这个方法论驱动
- **辅助**：提供支持，但不是主线
- **框架**：提供阶段结构，但具体执行由其他方法论驱动
- **参考**：可查阅但不主动使用

## 方法论协作关系

```
TOGAF           "做什么"
  │ 产出企业架构蓝图
  ↓
SDLC            "分几步做"
  │ 定义阶段和进出条件
  ↓
Agile/Scrum     "每步怎么跑"
  │ Sprint 迭代执行
  ↓
DevOps          "怎么交付"
  │ CI/CD 持续交付
  ↓
ITIL            "交付后怎么管"
  │ 服务管理和变更控制

Lean/Kanban ──── 贯穿所有阶段 ──── 持续优化流动效率
```

### 衔接点和交接物

| 衔接 | 上游交接物 | 下游接收 |
|------|---------|---------|
| TOGAF → SDLC | 企业架构蓝图、Gap-Fit 报告 | SDLC 各阶段的工件清单 |
| SDLC → Agile | 阶段进入条件达成、Sprint Backlog | Sprint 计划和执行 |
| Agile → DevOps | 代码提交、测试通过 | CI 流水线触发、部署执行 |
| DevOps → ITIL | 生产环境部署完成 | 事件管理、变更管理接管 |
| Lean → 所有 | 持续的流动效率数据 | 各方法论的优化改进 |

## 各方法论精简用法

### TOGAF ADM — "做什么"

- **采用**：ADM Preliminary + Phase A~D（架构愿景 / 业务架构 / 信息系统架构 / 技术架构）
- **不采用**：Phase E~H（实施治理），对 1 人 + AI 团队过重
- **建模工具**：ArchiMate（Archi 工具）
- **适用阶段**：Phase 1 主导
- **详见**：[togaf.md](./togaf.md)

### SDLC — "分几步做"

- **采用**：4 阶段模型（Phase 0~3），每阶段有明确进入/退出条件
- **特点**：Phase 2 内部嵌套 Agile Sprint（混合模式）
- **适用阶段**：全程框架
- **详见**：[sdlc.md](./sdlc.md) ✅ 已确认

### Agile/Scrum — "每步怎么跑"

- **采用**：Sprint（2~3 周）、Planning、Daily Check、Review、Retro
- **适配**：1 人 + AI 的精简 Scrum（无 Scrum Master 角色，开发者兼任）
- **适用阶段**：Phase 2 主导
- **详见**：[agile-scrum.md](./agile-scrum.md)

### DevOps — "怎么交付"

- **采用**：GitHub Actions CI/CD、Docker Compose 容器化、环境分层（local/dev/staging/prod）
- **不采用**：Kubernetes（项目规模不需要）
- **适用阶段**：Phase 0 搭建、Phase 2~3 持续运行
- **详见**：[devops.md](./devops.md)

### ITIL — "交付后怎么管"

- **采用**：事件管理、变更管理、服务级别管理（3 个核心流程）
- **不采用**：其余 ITIL 流程（过重）
- **适用阶段**：Phase 3 上线后
- **详见**：[itil.md](./itil.md)

### Lean/Kanban — "怎么更高效"

- **采用**：看板可视化（GitHub Projects）、WIP 限制、价值流分析
- **特点**：不独立使用，贯穿嵌入到其他方法论中
- **适用阶段**：全程辅助
- **详见**：[lean-kanban.md](./lean-kanban.md)
