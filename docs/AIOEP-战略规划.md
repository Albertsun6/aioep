# AIOEP 平台战略规划

> 状态: **已确认** | 版本: v1.0.0 | 创建: 2026-02-17
>
> 本文档是 AIOEP 平台的战略规划，定义平台的定位、目标、能力和演进策略。
> 区别于 ERP 战略规划（业务系统场景），本文档面向"平台产品"场景。

---

## 1. 平台定位

| 维度 | 内容 |
|------|------|
| 全称 | AIOEP — AI Organization Efficiency Platform（AI 组织效能平台） |
| 本质 | 将 AI-EADM 方法论体系软件化的平台 |
| 解决的问题 | AI 协助开发企业应用时，缺乏结构化的方法指导、状态追踪和知识传递 |
| 目标用户 | **当前**：自用（1人+AI）；**未来**：中小团队（1~5人+AI）做企业应用开发 |
| 核心价值 | 让 AI 不只是代码生成器，而是有方法论指导的"开发伙伴" |

### AIOEP 的本质公式

```
AIOEP = AI-EADM 方法论
      + 方法库（分析/开发/交付）
      + 模板库
      + Skills 引擎（AI 可执行的方法指令）
      + 项目上下文管理（AI 知道"做到哪了"）
      + 工具集成（Cursor / Archi / draw.io / GitHub）
```

### AIOEP 不是什么

- 不是项目管理工具（不替代 Jira/Linear）
- 不是代码生成平台（不替代 Cursor/Copilot）
- 不是 ERP 系统（ERP 是 AIOEP 的验证实例，不是 AIOEP 本身）

---

## 2. 平台愿景

### 1 年内（v0.1 ~ v0.3）

**"一个人 + AI，也能结构化地做出企业级应用"**

- AIOEP 以文件系统 + Cursor 为载体，让 AI 在每次对话中自动获得方法论指导
- 通过 Skills 引擎，AI 能按照方法步骤引导人类完成分析、设计、开发全流程
- 通过 project-context，AI 能跨对话保持上下文连续性

### 3 年内（v1.0+）

**"AI-EADM 成为企业应用开发的标准方法论"**

- AIOEP 成为可独立使用的开发效能平台
- 支持多项目管理、团队协作、方法论定制
- 开源社区贡献方法和 Skills

## 3. 驱动力分析

没有 AIOEP 的现状痛点（来自实际 ERP 项目开发经验）：

| 序号 | 痛点 | 表现 | 痛感 |
|------|------|------|------|
| D1 | **上下文丢失** | 每次新对话，AI 不知道项目做到哪了，需要人工恢复上下文 | **高** |
| D2 | **方法论无法执行** | 方法论写在文档里，但 AI 不会主动参考和遵循 | **高** |
| D3 | **知识不可传递** | 一个人积累的开发经验无法结构化地传递给另一个人+AI | **中** |
| D4 | **缺乏阶段导航** | 不知道"现在应该做什么"，需要人工判断当前阶段和下一步 | **中** |
| D5 | **方法模板散落** | 方法步骤和模板分布在多个文档中，AI 不知道去哪找 | **中** |
| D6 | **质量靠自觉** | 没有自动化的检查清单，质量门禁依赖人工记忆 | **低** |

## 4. 战略目标

| 编号 | 战略目标 | KPI | 目标值 | 时间线 |
|------|---------|-----|--------|--------|
| G1 | **验证 AI-EADM 方法论的可执行性** | 用 ERP 项目完整走完 Phase 1~3 | Phase 3 上线 | 2026 Q4 |
| G2 | **AI 上下文连续性** | 新对话恢复上下文所需人工操作数 | 0（全自动） | v0.1 |
| G3 | **方法可被 AI 执行** | AI 按方法步骤引导完成活动的成功率 | >= 80% | v0.1 |
| G4 | **Skills 引擎可用** | L1+L2 级别 Skills 覆盖 Phase 1 全部活动 | 100% 覆盖 | v0.2 |
| G5 | **多项目可复用** | 用 AIOEP 启动第二个项目的时间 | < 1 小时 | v0.3 |
| G6 | **方法论社区化** | AIOEP GitHub Stars | >= 100 | v1.0 |

> **G1 是首要目标**：方法论的价值只有通过实际项目验证才有意义。

## 5. 能力规划

AIOEP 需要的核心能力，按模块组织：

| 编号 | 能力 | 对应模块 | 优先级 | 解决痛点 |
|------|------|---------|--------|---------|
| C1 | 方法论结构化存储与导航 | M1 方法论内核 | **P0** | D2, D5 |

| C3 | 方法+模板按阶段导航 | M2 方法引擎 | **P0** | D4, D5 |
| C4 | AI 可执行的方法指令 | M4 Skills 引擎 | **P0** | D2, D3 |
| C5 | Cursor Rules/Skills 自动管理 | M5 Cursor 集成 | **P1** | D2 |
| C6 | 建模工具桥接 | M6 外部工具集成 | **P2** | — |
| C7 | 跨项目共享和复用 | M7 多项目管理 | **P2** | D3 |
| C8 | 可视化管理界面 | M8 Web 看板 | **P3** | D4 |

## 8. 约束与风险

| 约束/风险 | 描述 | 影响 | 应对策略 |
|---------|------|------|---------|
| **团队规模** | 1 人开发者 + AI | 所有工作量由一人承担 | MVP 优先，每个版本只做最小可用 |
| **Cursor 依赖** | v0.1~v0.2 完全依赖 Cursor 作为 UI | 平台能力受 Cursor 功能限制 | 设计时保持"文件系统优先"，不锁定 Cursor |
| **方法论未验证** | AI-EADM 未经完整项目验证 | 方法论可能需要大幅调整 | 用 ERP 项目逐步验证，快速迭代 |
| **Skills 有效性** | AI 能否按 Skill 指令稳定执行未知 | Skills 可能需要反复调优 | 从简单 Skill 开始，逐步增加复杂度 |
| **多仓库协作** | AIOEP 和项目仓库分离后的同步问题 | 方法更新需要手动同步到项目 | v0.1 手动同步，v0.2 CLI 自动化 |
| **过度设计** | 平台设计先于实际需求 | 浪费时间在不需要的功能上 | 严格 "ERP 需要什么就做什么" |

## 9. 演进策略

### 版本路线

```
v0.1 文件系统驱动（当前）
├── 方法论 + 方法 = 文件目录结构
├── Skills = .cursor/skills/ 下的 SKILL.md
├── 项目上下文 = .cursor/rules/project-context.md
├── UI = Cursor IDE
└── 验证: ERP Phase 1（需求分析与设计）

v0.2 + CLI 工具
├── aioep init → 从模板创建新项目
├── aioep sync → 同步 Skills/Rules 到项目
├── aioep status → 查看项目进度
└── 验证: ERP Phase 2 启动 + Skills 覆盖 Phase 1 全活动

v0.3 + Web 看板
├── 多项目状态看板
├── 方法导航 UI
├── Skills 在线管理
└── 验证: 第二个项目实例 + erp-portal 演进

v1.0 完整产品
├── 方法论定制化（用户可扩展 9 域）
├── Skills 市场（社区贡献）
├── 团队协作
└── 验证: 外部团队独立使用
```

### 开发-验证循环

每个 AIOEP 模块开发完成后，立即用 ERP 项目验证：

```
开发 AIOEP 模块 N → 用 ERP 验证模块 N → 回顾改进 → 开发模块 N+1
```

| AIOEP 模块 | ERP 验证场景 |
|------------|-------------|
| M1 方法论内核 | 方法论文档在 ERP Phase 1 中是否足够指导分析 |
| M3 项目上下文 | 新对话中 AI 是否自动知道 ERP 当前进度 |
| M2 方法引擎 | Gap-Fit 方法是否能被 AI 按步骤执行 |
| M4 Skills 引擎 | Phase 1 Skill 是否有效引导 ERP 需求分析 |
| M5 Cursor 集成 | Rules 是否正确约束 AI 在 ERP 开发中的行为 |
