# 统一术语表（Glossary）

> 基于 DDD 统一语言（Ubiquitous Language）原则，所有文档、代码和对话中使用统一术语。
> 新术语出现时必须更新本表。
>
> 版本: v0.1.0 | 最后更新: 2026-02-15

---

## 框架层级

| 术语 | 英文 | 定义 | 所属层级 |
|------|------|------|---------|
| AI-EADM | AI Enterprise Application Development Methodology | 本项目采用的通用方法论，适用于任何 AI 驱动的企业应用开发 | 方法论 |
| AI-EADM for Odoo | AI-EADM for Odoo | AI-EADM 方法论在 Odoo 平台上的具体实践方法 | 方法 |
| 方法论 | Methodology | 通用的、可复用的原则和框架体系。回答"做事的理论" | 方法论 |
| 方法 | Method | 方法论的平台特定化实例。回答"在某个平台上具体怎么做" | 方法 |
| 项目 | Project | 方法的一个具体应用实例。回答"当前做什么" | 项目 |
| 元结构 | Meta-Structure | 方法论的组成部分定义（9 域：原则/过程/建模/实践/质量/治理/AI/角色/工件） | 方法论 |
| 角色 | Role | 项目中的参与者类型，明确谁做什么 | 方法论 |
| 工件 | Artifact | 项目活动的有形产出（文档/模型/代码/测试/配置） | 方法论 |

## 设计模式与开发实践

| 术语 | 英文 | 定义 | 所属层级 |
|------|------|------|---------|
| TDD | Test-Driven Development | 测试驱动开发：先写测试 → 再写代码 → 重构 | 方法论 |
| BDD | Behavior-Driven Development | 行为驱动开发：用 Given/When/Then 规格化业务行为 | 方法论 |
| DDD | Domain-Driven Design | 领域驱动设计：用领域模型驱动系统设计 | 方法论 |
| EDA | Event-Driven Architecture | 事件驱动架构：通过事件解耦系统组件通信 | 方法论 |
| 限界上下文 | Bounded Context | DDD 概念，一个独立的业务边界。映射到 Odoo Module | 方法论 |
| 聚合根 | Aggregate Root | DDD 概念，聚合的入口实体。映射到 Odoo 主模型 | 方法论 |
| 统一语言 | Ubiquitous Language | DDD 概念，团队内使用统一的业务术语。即本术语表 | 方法论 |
| Gap-Fit 分析 | Gap-Fit Analysis | 将需求与框架标准功能逐项对比，找出差距和匹配 | 方法论 |

## 维护规则

1. **新术语出现时**：立即添加到本表对应分类中
2. **术语含义变更时**：更新定义并在变更记录中说明
3. **AI 对话中**：使用本表中的标准术语，避免同义替换
4. **代码命名中**：变量/函数/模块名与术语表保持一致
