# 培训交接方法

> 状态: **已验证** | 版本: v0.3.0 | 创建: 2026-02-16 | 上次验证: 2026-02-17（ERP 项目 Phase 3 试跑）

## 定位

指导如何完成用户培训、操作手册编写、运维手册编写、观察期运行和正式交接。对应 SDLC Phase 3 的第三大活动，是项目交付的最后环节。

适用于 Odoo 17 Community 项目，1 人 + AI 开发模式下的精简交接策略。核心目标：**让用户能独立使用系统，让运维人员能独立维护系统**。

## 前置条件

- 系统已上线运行（部署迁移方法完成）
- 上线后健康检查通过（无 P0/P1 问题）
- 用户账号和角色权限已配置完成
- 培训环境准备就绪（可以是生产环境的只读副本或独立 sandbox）

## 步骤

### Step 1: 用户培训

**做什么**：按角色分组进行系统操作培训，确保每个角色的使用者能独立完成日常操作。

**AI 引导问题**：
1. 有哪些角色需要培训？每个角色有多少人？
2. 培训方式是集中培训还是分批？线下还是线上？
3. 是否有操作基础？（从 Odoo 14 迁移的用户可能有基础）
4. 培训时间窗口是多少？（建议每个角色 2-4 小时）

**操作要点**：
- **按角色分组**：不同角色只培训与自己工作相关的功能
- **培训方式**：演示 + 跟练 + 独立操作（1 人项目可录屏替代集中培训）
- **AI 辅助生成培训材料**：
  - 利用 AI 根据 BDD 行为规格生成操作步骤说明
  - 利用截图工具批量生成操作截图
  - 生成 FAQ 文档（基于 UAT 阶段的常见问题）
- **培训考核**：每个角色完成 3-5 个关键操作场景即为通过
- **培训反馈收集**：培训结束后收集用户反馈，记录改进点

> **1 人 + AI 项目的培训模式**（试跑经验）：
> - 培训材料由 AI 基于 BDD 行为规格和 Odoo 操作路径自动生成
> - 操作手册结构：角色 → 日常操作 → 周期操作 → 异常处理，每个操作含前置条件 + 步骤 + 预期结果
> - 试跑中为采购角色生成了完整操作手册（含 RFQ 创建、PO 确认、审批操作、收货、对账），实测覆盖核心场景
> - 运维手册可包含"快速命令参考"附录（`docker` 常用命令、备份恢复一键命令），降低运维学习曲线
> - 1 人项目的"培训考核"可转化为"自我检查清单"——按操作手册步骤独立走一遍，确认无遗漏

**培训角色-内容矩阵**：

| 角色 | 培训模块 | 关键操作 | 预计时长 |
|------|---------|---------|---------|
| **系统管理员** | 全模块 + 系统设置 | 用户管理、权限配置、系统参数、数据导入导出 | 4h |
| **采购员** | 采购管理 | 创建 RFQ → 确认 PO → 收货 → 对账 | 3h |
| **采购经理** | 采购管理 + 审批 | 审批 PO、供应商管理、采购报表 | 2h |
| **销售员** | 销售管理 | 创建报价单 → 确认 SO → 发货 → 开票 | 3h |
| **仓库管理员** | 库存管理 | 入库/出库操作、库存调整、库存盘点 | 3h |
| **财务人员** | 财务管理 | 供应商账单、客户发票、付款收款、对账 | 3h |

**输出**：培训记录（填入模板 Section 1）

### Step 2: 操作手册

**做什么**：为每个角色编写操作手册，作为日常工作的参考文档。操作手册是培训的持久化产物。

**AI 引导问题**：
1. 手册是电子版还是纸质版？（建议电子版 Markdown，便于更新）
2. 每个操作是否需要截图？（强烈建议，降低理解成本）
3. 手册放在哪里让用户方便访问？（项目 Wiki / 共享文件夹 / Odoo 内置知识库）

**操作要点**：
- **按角色分册**：每个角色一个操作手册，只包含该角色需要的操作
- **统一格式**：每个操作包含以下结构：
  - 操作名称
  - 前置条件（需要什么数据/权限）
  - 操作步骤（编号列表 + 截图）
  - 预期结果（操作完成后应该看到什么）
  - 常见问题（FAQ）
- **AI 辅助生成**：
  - 基于 BDD 行为规格自动生成操作步骤框架
  - 使用浏览器自动化工具批量截图
  - AI 审查手册的准确性和完整性
- **Odoo 内置帮助**：利用 Odoo 的 tooltip 和帮助文本补充关键字段说明
- **版本管理**：手册纳入 Git 版本控制，随模块升级同步更新

> **AI 生成操作手册的效率**（试跑经验）：试跑中 AI 基于 Gap-Fit 报告中的需求描述 + Odoo 标准操作路径，一次性生成了采购操作手册（约 200 行 Markdown），涵盖 5 个主要操作场景。AI 也生成了运维手册（约 300 行），涵盖架构概述、日常检查、备份恢复、故障排查和快速命令参考。质量可用，仅需人工微调场景描述。建议在 Phase 2 每个 Sprint 完成后即更新操作手册，避免集中到 Phase 3 才补写。

**操作手册结构模板**：

```
# [角色] 操作手册

## 1. 日常操作
### 1.1 [操作名称]
**前置条件**: ...
**步骤**:
1. 进入菜单 [截图]
2. 点击 [按钮名] [截图]
3. 填写字段 [截图]
4. 确认提交 [截图]
**预期结果**: ...
**FAQ**: ...

## 2. 周期性操作
### 2.1 [操作名称]
...

## 3. 异常处理
### 3.1 [场景]
...
```

**输出**：各角色操作手册（按实际角色数量）

### Step 3: 运维手册

**做什么**：编写系统运维手册，涵盖日常维护、监控告警、备份恢复、故障排查等运维 SOP。

**AI 引导问题**：
1. 运维由谁负责？（开发者本人 / IT 部门 / 外包）
2. 运维人员的技术水平？（决定手册的详细程度）
3. 有没有现有的运维工具链？（监控、告警、日志平台）

**操作要点**：
- **日常维护清单**（每日/每周/每月）：
  - 每日：检查服务状态、查看错误日志、确认备份完成
  - 每周：磁盘空间清理、数据库 vacuum、日志归档
  - 每月：备份恢复测试、SSL 证书有效期检查、安全补丁检查
- **监控告警处理 SOP**：
  - 每种告警对应的处理步骤
  - 升级路径（何时联系开发者）
- **备份恢复 SOP**：
  - 完整的备份恢复步骤（含命令行）
  - 恢复验证清单
  - 演练频率（建议每月 1 次）
- **常见问题排查**：
  - Odoo 服务无法启动 → 检查日志 → 常见原因清单
  - 数据库连接失败 → 检查步骤
  - 页面 500 错误 → 排查路径
  - 性能下降 → 检查项

**运维手册目录结构**：

```
# 运维手册

## 1. 系统架构概述
   - 服务器信息、容器架构、网络拓扑

## 2. 日常维护
   - 2.1 每日检查清单
   - 2.2 每周维护任务
   - 2.3 每月维护任务

## 3. 监控与告警
   - 3.1 监控指标说明
   - 3.2 告警处理 SOP

## 4. 备份与恢复
   - 4.1 自动备份配置
   - 4.2 手动备份步骤
   - 4.3 恢复步骤

## 5. 故障排查
   - 5.1 服务无法启动
   - 5.2 数据库异常
   - 5.3 性能问题
   - 5.4 模块升级失败

## 6. 安全维护
   - 6.1 密码策略
   - 6.2 SSL 证书更新
   - 6.3 安全补丁更新

## 7. 应急联系
   - 联系人、联系方式、响应时间
```

**输出**：运维手册

### Step 4: 观察期

**做什么**：上线后进入 1 周观察期，每日跟踪系统运行指标和用户反馈，确保系统稳定运行。

**AI 引导问题**：
1. 观察期多长合理？（建议 1 周，复杂系统可延长至 2 周）
2. 观察期的值班安排？
3. 观察期结束的退出条件是什么？

**操作要点**：
- **每日检查项**：
  - 系统可用率（目标 ≥ 99%）
  - P0/P1 缺陷数（目标：P0 = 0，P1 ≤ 1）
  - 用户反馈收集和分类
  - 错误日志趋势
  - 关键业务指标（订单处理数、响应时间）
- **问题处理优先级**：
  - P0：立即修复（< 4 小时）
  - P1：当日修复（< 8 小时）
  - P2：观察期内修复
  - P3：纳入后续迭代
- **每日简报**：向相关干系人发送一句话状态更新（1 人项目可记录在 Chatlog 中）

> **1 人项目的观察期简化**（试跑经验）：试跑验证了观察期指标跟踪表的可行性。1 人项目可将观察期简化为：(1) 每日 5 分钟健康检查脚本（自动化）、(2) 每日查看错误日志（`docker logs --since 24h`）、(3) Chatlog 记录每日状态。观察期退出条件保持不变。

- **观察期退出条件**：
  - 连续 3 天无 P0/P1 缺陷
  - 系统可用率 ≥ 99%
  - 核心业务流程无阻塞
  - 用户满意度反馈正面

**观察期指标跟踪表**：

| 指标 | Day 1 | Day 2 | Day 3 | Day 4 | Day 5 | Day 6 | Day 7 |
|------|-------|-------|-------|-------|-------|-------|-------|
| 系统可用率 | | | | | | | |
| P0 缺陷数 | | | | | | | |
| P1 缺陷数 | | | | | | | |
| P2 缺陷数 | | | | | | | |
| 用户反馈数 | | | | | | | |
| 错误日志数 | | | | | | | |

**输出**：观察期报告（填入模板 Section 2）

### Step 5: 正式交接

**做什么**：完成所有交接物的移交，明确运维责任，签署 SLA，正式结束项目实施阶段。

**AI 引导问题**：
1. 交接给谁？（内部 IT / 外包运维 / 开发者继续维护）
2. SLA 如何定义？（响应时间、解决时间、可用率承诺）
3. 后续支持期多长？（建议 1-3 个月过渡支持）

**操作要点**：
- **交接物清单**（逐项确认移交）：
  - 账号密码：管理员账号、数据库密码、服务器 SSH 密钥、SSL 证书
  - 代码仓库：Git 仓库访问权限、分支说明、CI/CD 配置
  - 文档：操作手册、运维手册、架构设计文档、API 文档
  - 环境：生产/staging/开发环境的访问信息
  - 第三方服务：域名注册商、SSL 提供商、云服务账号
- **运维责任矩阵**：明确谁负责什么

| 职责 | 日常运维 | 问题修复 | 功能迭代 | 安全更新 |
|------|---------|---------|---------|---------|
| 负责人 | [姓名/团队] | [姓名/团队] | [姓名/团队] | [姓名/团队] |
| 响应时间 | [N]h | [N]h | [N] 天 | [N]h |

- **SLA 定义**：

| 等级 | 定义 | 响应时间 | 解决时间 |
|------|------|---------|---------|
| **紧急** | 系统不可用 / 数据丢失 | ≤ 1h | ≤ 4h |
| **高** | 核心功能异常 / 无绕过方案 | ≤ 4h | ≤ 8h |
| **中** | 功能异常 / 有绕过方案 | ≤ 8h | ≤ 3 天 |
| **低** | 体验优化 / 非紧急需求 | ≤ 24h | 下次迭代 |

- **正式确认**：
  - 交接清单逐项签字确认
  - SLA 双方签字
  - 明确过渡支持期和退出时间

**输出**：交接确认书（填入模板 Section 3）

## 培训角色-内容定义

| 角色 | 系统权限范围 | 必须掌握的操作 | 考核标准 |
|------|------------|--------------|---------|
| **系统管理员** | 全部 | 用户管理、系统配置、数据导入导出、备份恢复 | 独立完成用户创建 + 权限配置 + 数据导入 |
| **采购员** | 采购模块 | RFQ → PO → 收货 → 对账 | 独立完成一笔完整采购流程 |
| **采购经理** | 采购模块 + 审批 | PO 审批、供应商管理、采购报表 | 独立完成审批 + 导出报表 |
| **销售员** | 销售模块 | 报价单 → SO → 发货 → 开票 | 独立完成一笔完整销售流程 |
| **仓库管理员** | 库存模块 | 入库、出库、调拨、盘点 | 独立完成入库 + 盘点 |
| **财务人员** | 财务模块 | 账单、发票、付款、收款、对账 | 独立完成付款 + 对账 |

| 分类 | 交接物 | 格式 | 存放位置 |
|------|--------|------|---------|
| **账号密码** | 管理员账号 | 加密文件 | 密码管理器 |
| | 数据库密码 | 加密文件 | 密码管理器 |

| | 运维手册 | Markdown | 仓库 `docs/` |
| | 架构设计 | Markdown | 仓库 `docs/` |
| **环境** | 生产环境信息 | 文档 | 加密存储 |
| | staging 环境信息 | 文档 | 加密存储 |
| **第三方** | 域名注册信息 | 文档 | 加密存储 |
| | 云服务账号 | 文档 | 密码管理器 |

## 检查清单

- [ ] 所有角色的培训已完成，培训记录已归档
- [ ] 每个角色至少通过 3-5 个关键操作场景考核
- [ ] 各角色操作手册已编写完成
- [ ] 运维手册已编写完成（含日常维护 + 故障排查 + 备份恢复）
- [ ] 观察期已完成（≥ 1 周），退出条件全部满足
- [ ] 观察期内无遗留 P0/P1 缺陷
- [ ] 交接物清单逐项确认移交
- [ ] SLA 已定义并双方确认
- [ ] 运维责任矩阵已明确
- [ ] 正式交接书已签署

- 培训记录（含签到、考核结果、反馈）
- 各角色操作手册
- 运维手册
